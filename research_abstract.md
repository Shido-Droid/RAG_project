# 卒業研究 作品要旨

## 1. 作品コンセプト
本研究は、Retrieval-Augmented Generation (RAG) 技術の学習と実践を目的として行われた。
成果物として、学生が日々の学習や研究活動において直面する「情報の探索と整理」を支援するためのWebアプリケーションを作成した。
既存のLLMチャットボットでは情報の鮮度や根拠の不透明さが課題となるが、本Webアプリは最新情報の取得とローカル資料の活用を組み合わせることで、より信頼性の高い学習支援ツールとなることを目指した。

## 2. 作品説明
本アプリケーションは、大学のクラスメートなどの学生を利用対象とした検索・質問応答システムである。
最大の特徴は、ユーザーが保有する「ローカル環境のドキュメント（PDF, Word, etc.）」と、インターネット上の「Web検索（最新情報）」をハイブリッドで活用できる点にある。
これにより、授業配布資料などのクローズドな情報と、ニュースや技術記事などのオープンで最新の情報を横断的に検索し、統合された回答を生成することが可能となっている。

## 3. 作成環境・使用技術
限られた計算資源で効率的に動作させるため、軽量かつ高速な技術選定を行った。

*   **OS**: Linux (Ubuntu)
    *   サーバーおよび開発環境のホストOSとして使用。
*   **開発環境**: 
    *   **Visual Studio Code**: ソースコードの編集、デバッグ、Git操作を行う統合開発環境として使用。
    *   **Git**: ソースコードのバージョン管理に使用。
    *   **Docker**: アプリケーションのコンテナ化と環境構築の再現性確保のために使用。
    *   **Antigravity (AI Agent)**: Google DeepMindが開発したAIコーディングアシスタント。設計、実装、リファクタリング、ドキュメント作成のペアプログラミングパートナーとして活用。
*   **LLM基盤**: LM Studio
    *   ローカル環境でLLM (**Qwen2.5 7B**) を動作させ、OpenAI互換のAPIサーバーを提供するために使用。
*   **バックエンド**: Python, FastAPI
    *   **Python**: バックエンドロジックの主要言語として使用。
    *   **FastAPI**: 高速な非同期処理が可能なWeb APIフレームワークとして、リクエスト処理やストリーミング応答の実装に使用。
    *   **ChromaDB**: アップロードされた文書をベクトル化して保存し、高速な類似検索を行うベクトルデータベースとして使用。
    *   **DuckDuckGo Search API**: 最新情報を取得するためのWeb検索エンジンとして使用。
    *   **BeautifulSoup4 / Trafilatura**: 検索結果のWebページからテキスト本文を抽出（スクレイピング）するために使用。
    *   **Sentence-Transformers**: 日本語テキストをベクトル（埋め込み表現）に変換するために使用。
    *   **PyPDF / Python-docx / Pytesseract**: 各種ドキュメントファイルからのテキスト抽出およびOCR処理に使用。
*   **フロントエンド**: React, Vite
    *   **React**: ユーザーインターフェースの構築に使用。
    *   **Vite**: 高速なフロントエンドビルドツールおよび開発サーバーとして使用。
    *   **Tailwind CSS**: 効率的なスタイリングとGlassmorphismデザインの実装に使用。
    *   **Marked / Highlight.js**: AIの回答に含まれるMarkdownやコードブロックを整形して表示するために使用。

## 4. 工夫点
*   **低スペックマシンでのRAG精度向上**:
    GPUリソースが限られた環境でも実用的な応答速度と精度を出すため、検索結果のRe-ranking（再順位付け）処理や、コンテキストサイズの最適化を行った。
*   **直感的なUIデザイン**:
    既存のチャットボットのような無機質なデザインではなく、Glassmorphismを取り入れた「Premium」なデザインを採用し、学生が使いたくなるようなUXを目指した。
*   **ハイブリッド検索ロジック**:
    ユーザの質問意図（Intent）を判定し、Web検索が必要な場合とローカル文書のみで完結する場合を自動で切り替えるロジックを実装した。

## 5. 改善点・今後の課題
*   **応答速度の改善**:
    ローカルLLMとWeb検索を組み合わせているため、回答生成までに時間を要する場合がある。ストリーミング処理は実装したが、ファーストトークンまでの時間短縮が課題である。
*   **マルチモーダル対応**:
    現在は主にテキストデータを扱っているが、図表や画像を含む資料からの高度な情報抽出には改善の余地がある。
*   **出典の明記と検証**:
    回答の根拠となるソース元の提示機能はあるが、より詳細な引用（Page番号など）の精度向上が求められる。

## 6. まとめ
本研究を通じて、RAGシステムの構築に必要な一連の技術（ベクトル検索、LLMプロンプトエンジニアリング、Web開発）を習得することができた。
また、単に動くものを作るだけでなく、ターゲットユーザー（学生）にとって使いやすいUI/UXや、限られたリソースでのパフォーマンス最適化など、実運用を意識した開発経験を積むことができた点は非常に有意義であった。
この経験は今後のエンジニアリング活動において大きな糧になると確信している。
